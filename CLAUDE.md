# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`codebuilder` is a CrewAI **Flow** (not a plain Crew — see `[tool.crewai] type = "flow"` in `pyproject.toml`) that ingests a project brief, plans it with human-in-the-loop (HITL) approval, and then builds code in a per-job workspace. Async HITL pause/resume works because CrewAI auto-creates a `SQLiteFlowPersistence()` when `HumanFeedbackPending` is raised, and `CodebuilderFlow.from_pending(job_id).resume(feedback)` reads from that same default DB. `@persist()` on the flow class is intentionally NOT applied — we don't use per-step crash recovery, and skipping it avoids writing a state snapshot after every step of a one-shot job.

`AGENTS.md` in the repo root is the canonical CrewAI reference (auto-generated by `crewai create`). Its "Mandatory: Research before writing CrewAI code" section applies to any changes to agents/crews/tools/flows — follow it.

## Commands

Dependency management uses `uv` (see `uv.lock`). Python must be `>=3.10, <3.14`.

```bash
uv sync                        # install deps
uv add <pkg>                   # add a dep (don't hand-edit pyproject)

# Entrypoints (defined in pyproject [project.scripts]; all map to src/codebuilder/main.py)
uv run kickoff                 # start a new job with the hardcoded test inputs in main.py::kickoff()
uv run plot                    # render codebuilder_flow.html (flow graph)
uv run run_crew                # alias for kickoff

crewai run                     # same as `uv run kickoff`
```

Lint / test use `ruff` and `pytest` via the in-repo `LintRunnerTool` / `TestRunnerTool`. To run them directly:

```bash
uv run ruff check src
uv run pytest -q
uv run pytest -q tests/test_foo.py::test_name   # single test
```

## Architecture

### Flow (src/codebuilder/main.py)

`CodebuilderFlow` is a single `Flow[CodebuilderState]` with five methods wired by decorators:

1. `ingest` (`@start`) — relies on CrewAI Flow auto-merging top-level `inputs={...}` keys into `self.state` (so `session_id`, `brief`, `project_name`, `goals`, `tech_stack`, `attachments` are already populated when `ingest` runs). Coerces attachment dicts → `Attachment`, creates `workspaces/<session_id|flow_id>/{inputs,output}`, materializes attachments, and derives `state.project_key` via `history.project_key_from(...)`.

   **Two distinct identifiers, do not collapse them.**
   - `state.session_id` (caller-supplied) is the canonical *user/UI* identifier. Workspace dir, project_key fallback, S3 prefix, progress-webhook `session_id` field, and HITL-webhook `session_id` field all key off it. The frontend uses it as the URL slug and to correlate inbound webhook events back to the right session.
   - `state.id` (= `flow_id`, auto-generated UUID by `Flow.__init__`) is the *flow execution* identifier. History rows (one per execution), the S3 per-execution segment, and `Flow.from_pending(flow_id).resume(...)` all use it.

   **Never pass `id` in kickoff inputs.** CrewAI Flow auto-merge will overwrite `state.id` *before* OTel emits its first span. AMP fetches traces from Wharf using the *overridden* state.id but Wharf has them indexed under the *original* flow_id, so traces from the pre-resume phase (ingest, plan, planner-crew tasks) are stranded. After resume the flow is reconstructed with `state.id` already set, so post-resume traces correlate, which produces the symptom "AMP traces only show after plan approval" (filed upstream as Linear CON-101 / COR-48 / CON-56).

   **Do not** mutate `CREWAI_STORAGE_DIR` here (or anywhere at runtime): crewai uses that env var as the `app_name` input to `appdirs.user_data_dir(...)`, which determines where `SQLiteFlowPersistence()` writes pending-feedback rows. `Flow.from_pending(flow_id)` always constructs a default `SQLiteFlowPersistence()`, so drift between save-time and resume-time breaks HITL resume with "No pending feedback found for flow_id" (AMP hit this).
2. `plan` (`@listen(ingest)` + `@human_feedback`) — runs `PlannerCrew`, then pauses for human review. The `@human_feedback` decorator emits one of `approved | amend | rejected`.
3. `revise_plan` (`@listen("amend")`) — loops the plan back through `PlannerCrew` with prior plan + user amendments, re-gating on another `@human_feedback`.
4. `build` (`@listen("approved")`) — chooses `build_dir` based on `plan.mode` (`patch_existing` writes into the cloned repo under `inputs/repo`; `new_project` writes into `output/` and `git init`s it), then loops over `plan.subtasks` running the Writer→Reviewer loop.
5. `finalize` (`@listen(build)`) — runs deterministic integration QA (`ruff` + `pytest`) over the whole workspace. If QA fails, it gives `WriterCrew.repair_crew()` one repair pass by default, reruns QA, then runs the domain architecture gate for new-project jobs via `run_full_architecture_gate(build_dir, plan)`. The gate is dispatched on `plan.domain` (a slug the planner emits, matching an activated architecture-defining skill); `runtime_qa._ARCHITECTURE_GATES` maps slugs to gate functions (currently only `"rpa"`). Projects with no domain — or a domain without a registered gate — finalize on lint/test alone. `finalize` uploads artifacts, returns a completion payload, captures a git diff for patch jobs, and writes a `project_history` row via `history.record(...)`. `on_rejected` and build-failure branches also call `history.record` so every terminated job is auditable.

State is a pydantic `CodebuilderState` (`schemas.py`) extending `FlowState`. `JobStatus` transitions: `pending → planning → awaiting_approval → executing → done|failed`.

**HITL wiring.** `feedback_provider.WebhookFeedbackProvider` POSTs to `$CODEBUILDER_APPROVAL_WEBHOOK` and raises `HumanFeedbackPending` to pause the flow. If the env var is unset it falls back to `ConsoleProvider` so the flow still works in a terminal. Resumes always go through `CodebuilderFlow.from_pending(job_id).resume(feedback)`.

### Crews (src/codebuilder/crews/)

Three `@CrewBase` classes, each with YAML-driven `agents.yaml` + `tasks.yaml` siblings:

- `PlannerCrew` — two agents (`planner` on Opus, `planner_lite` on Sonnet) feeding a 3-task chain (`skeleton_task` → `review_skeleton_task` → `expand_task`) wired via CrewAI `Task(context=[…])`. The lead `planner` runs only `skeleton_task` (the architectural decision step); `planner_lite` runs the mechanical review and expand passes. Final crew result is the `Plan` from `expand_task` (pydantic-validated, guardrail enforces 1–60 subtasks with non-empty `file_path`/`test_criteria`). Both agents have only `FileReadTool` + `DirectoryReadTool` — neither writes files. Both use the `rpa` CrewAI skill. The `skeleton_task` prompt receives a `prior_history` markdown block assembled from the `project_history` SQLite table; `expand_task` does not (the reviewed skeleton already carries forward the structural decisions, no point re-paying the token cost).
- `WriterCrew(workspace_dir)` — one `writer` agent. `.crew()` runs `write_task` for planned files; `.repair_crew()` runs `repair_task` after failed final QA and patches one failing file. Scoped to `workspace_dir` via the three `Workspace*Tool`s. Deterministic review forbids placeholder/TODO/`...` stubs.
- `ReviewerCrew(workspace_dir)` — `.crew()` runs per-file review fallback, `.qa_crew()` provides LLM QA diagnostics, and `.architecture_gate_crew()` is the domain architecture acceptance gate for new projects (driven by `plan.domain`). Reviewer/QA agents use both the `rpa` and `code-review-gate` skills. The deterministic final-QA path still runs `ruff` + `pytest` first.

The Writer↔Reviewer loop in `main.py::_build_subtask` retries up to `CODEBUILDER_MAX_SUBTASK_RETRIES + 1` times per subtask, feeding prior review issues back into the writer's inputs. Final QA repair is controlled separately by `CODEBUILDER_MAX_FINAL_QA_REPAIRS` (default `1`). Skipped lint/tests are failures. New-project jobs whose plan declares a `domain` registered in `_ARCHITECTURE_GATES` must also pass that gate before status becomes `done`.

### Schemas (src/codebuilder/schemas.py)

All crew outputs go through pydantic models: `Plan`, `SubTask`, `CodeArtifact`, `ReviewResult`, `QAReport`, `Attachment`. When adding fields, the YAML task description's `Rules:` section and `expected_output:` typically need matching updates, since the agent produces JSON against this schema.

`CodebuilderState` also carries `project_name` (user-supplied, for display and `new_project` keying) and `project_key` (derived stable identifier used for the `project_history` table). `Plan.project_name` stays as the planner's agent-produced display name — distinct from the state field.

### Per-project history (src/codebuilder/history.py)

Standalone SQLite table `project_history` at `$CODEBUILDER_HISTORY_DB` (defaults to `./data/codebuilder_history.db`), independent of CrewAI's flow persistence (only the HITL pending-feedback rows that CrewAI's auto-fallback writes live there). One row per terminated job keyed by `(project_key, job_id)`. Public API:

- `canonicalize_git_url(url)` — reduces git URLs to `host/org/repo` (handles HTTPS/SSH/scp-style/casing/`.git` suffix).
- `project_key_from(state)` — derives the stable key: canonical git URL from the first `Attachment` with `kind == "git"`, else slug of `project_name`, else `""` (main.py falls back to `flow_id`).
- `record(state)` — upserts a row on `finalize` / `on_rejected` / build-failure. Serialises `Plan`, `QAReport`, artifact `file_path`s, aggregated reviewer issues, and git diff (for patch jobs).
- `summarize_for_planner(project_key, limit=3)` — compact markdown of the last N runs (date, mode, status, files touched, top reviewer issues, QA notes) fed into `{prior_history}` in the planner task YAML. Returns `""` when there's no history.

History is observability, not correctness — all call sites wrap `history.record(...)` in `try/except` so a DB failure never breaks the flow.

### Tools (src/codebuilder/tools/)

- `workspace_tool.py` — `WorkspaceRead/Write/List` all share `resolve_within()` which enforces that agent-supplied relative paths cannot escape the job workspace. All file I/O from agents goes through these; never give agents raw `FileReadTool` pointed at a real filesystem path.
- `lint_runner_tool.py` — shells out to `ruff check` / `pytest -q` inside the workspace (120s / 300s timeouts). Returns `"PASS"` on clean exit, otherwise the captured stdout+stderr.
- `git_tool.py` — thin `gitpython` wrappers: `clone`, `init_and_commit`, `diff`. Used for patch-mode diff extraction and new-project baselines.
- `attachment_tool.py::materialize()` — decodes the `Attachment` list (git URLs cloned, zips extracted, PDFs saved + text-extracted via `pypdf`, images saved raw) into `workspace/inputs/`. Cloned repos end up at `inputs/repo`, which `build` in `main.py` looks up for `patch_existing` jobs.

### Skills

Methodology and checklist guidance lives in CrewAI skills under `src/codebuilder/skills/`: `rpa` carries the canonical RPA onboarding guide (Portuguese, full process/architecture/SOLID/TDD/folder-layout standard) and `code-review-gate` carries the domain-agnostic acceptance criteria for generated projects. Each crew file pre-discovers the skills dir into a `_SKILLS` dict at module load and passes specific entries inline (`skills=[_SKILLS["rpa"]]`), the same way tools are passed. New domain skills (e.g. `python-package`, `flask-api`) drop into `skills/` and get attached by name. There is no separate skill loader and no `knowledge/` directory — `Agent.skills` accepts pre-loaded `Skill` objects directly and `Agent.set_skills` activates them with proper event emission. Skill slugs are also the keys used by the architecture-gate registry (`_ARCHITECTURE_GATES` in `runtime_qa.py`) when the planner emits a matching `plan.domain`.

## Conventions specific to this repo

- Workspace root is `$CODEBUILDER_WORKSPACE_ROOT` (defaults to `./workspaces`). Each job lives in `<root>/<flow_id>/` with `inputs/` and `output/` subdirs. Never let tools touch anything outside this root.
- Cross-run context comes from the `project_history` SQLite table. Do not mutate `CREWAI_STORAGE_DIR` at runtime (HITL resume depends on it being stable).
- The stable project identifier is `state.project_key`, not `state.project_name`. Always derive via `history.project_key_from(state)` — never reconstruct ad-hoc from the name (it loses the git-URL canonicalisation that makes two users hitting the same repo share context).
- Kickoff payloads MUST use `session_id` (not `id`) for the caller's session identifier. See the `ingest` note for why — overriding `state.id`/`flow_id` strands AMP OTel traces. The HITL feedback webhook and the progress webhook both emit a `session_id` field; the frontend correlates incoming events on `session_id` first, with `flow_id`/`job_id` as fallbacks.
- Default models and planning settings live in each crew's `agents.yaml`. Edit YAML for agent model changes. The only remaining model env override is `CODEBUILDER_GUARDRAIL_LLM` for the `@human_feedback` classifier.
- Planner YAML sets `planning: true` on the lead `planner` (skeleton step) only. `planner_lite` and the writer run without `planning`/`reasoning` — the writer follows an explicit, fully-specified subtask and doesn't need a reasoning pre-pass.
- When adding a task that returns structured data, set `output_pydantic=<Schema>` and, where correctness matters, a string `guardrail=...` on the `Task`. See `skeleton_task` / `expand_task` / `write_task` for examples.
- `_planner_inputs()` in `main.py` is the single source of truth for the dict passed to `PlannerCrew`. Add new template variables here **and** in `crews/planner_crew/config/tasks.yaml` together — the YAML uses `{var_name}` interpolation and will raise on missing keys.
- Kickoff payloads accept an optional top-level `project_name` (required for useful `project_history` keying in `new_project` mode; `patch_existing` jobs derive the key from the git attachment URL instead).
