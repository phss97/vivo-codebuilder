# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

`codebuilder` is a CrewAI **Flow** (not a plain Crew ‚Äî see `[tool.crewai] type = "flow"` in `pyproject.toml`) that ingests a project brief, plans it with human-in-the-loop (HITL) approval, and then builds code in a per-job workspace. Async HITL pause/resume works because CrewAI auto-creates a `SQLiteFlowPersistence()` when `HumanFeedbackPending` is raised, and `CodebuilderFlow.from_pending(job_id).resume(feedback)` reads from that same default DB. `@persist()` on the flow class is intentionally NOT applied ‚Äî we don't use per-step crash recovery, and skipping it avoids writing a state snapshot after every step of a one-shot job.

`AGENTS.md` in the repo root is the canonical CrewAI reference (auto-generated by `crewai create`). Its "Mandatory: Research before writing CrewAI code" section applies to any changes to agents/crews/tools/flows ‚Äî follow it.

## Commands

Dependency management uses `uv` (see `uv.lock`). Python must be `>=3.10, <3.14`.

```bash
uv sync                        # install deps
uv add <pkg>                   # add a dep (don't hand-edit pyproject)

# Entrypoints (defined in pyproject [project.scripts]; all map to src/codebuilder/main.py)
uv run kickoff                 # start a new job with the hardcoded test inputs in main.py::kickoff()
uv run plot                    # render codebuilder_flow.html (flow graph)
uv run run_crew                # alias for kickoff

crewai run                     # same as `uv run kickoff`
```

Lint / test use `ruff` and `pytest` via the in-repo `LintRunnerTool` / `TestRunnerTool`. To run them directly:

```bash
uv run ruff check src
uv run pytest -q
uv run pytest -q tests/test_foo.py::test_name   # single test
```

## Architecture

### Flow (src/codebuilder/main.py)

`CodebuilderFlow` is a single `Flow[CodebuilderState]` with five methods wired by decorators:

1. `ingest` (`@start`) ‚Äî relies on CrewAI Flow auto-merging top-level `inputs={...}` keys into `self.state` (so `session_id`, `brief`, `project_name`, `goals`, `tech_stack`, `attachments` are already populated when `ingest` runs). Coerces attachment dicts ‚Üí `Attachment`, creates `workspaces/<session_id|flow_id>/{inputs,output}`, materializes attachments, and derives `state.project_key` via `history.project_key_from(...)`. For `patch_existing`, it resolves the attached project root when possible, runs one deterministic preflight QA pass, and stores the result in `state.preflight_qa_report` for planner context.

   **Two distinct identifiers, do not collapse them.**
   - `state.session_id` (caller-supplied) is the canonical *user/UI* identifier. Workspace dir, project_key fallback, S3 prefix, progress-webhook `session_id` field, and HITL-webhook `session_id` field all key off it. The frontend uses it as the URL slug and to correlate inbound webhook events back to the right session.
   - `state.id` (= `flow_id`, auto-generated UUID by `Flow.__init__`) is the *flow execution* identifier. History rows (one per execution), the S3 per-execution segment, and `Flow.from_pending(flow_id).resume(...)` all use it.

   **Never pass `id` in kickoff inputs.** CrewAI Flow auto-merge will overwrite `state.id` *before* OTel emits its first span. AMP fetches traces from Wharf using the *overridden* state.id but Wharf has them indexed under the *original* flow_id, so traces from the pre-resume phase (ingest, plan, planner-crew tasks) are stranded. After resume the flow is reconstructed with `state.id` already set, so post-resume traces correlate, which produces the symptom "AMP traces only show after plan approval" (filed upstream as Linear CON-101 / COR-48 / CON-56).

   **Do not** mutate `CREWAI_STORAGE_DIR` here (or anywhere at runtime): crewai uses that env var as the `app_name` input to `appdirs.user_data_dir(...)`, which determines where `SQLiteFlowPersistence()` writes pending-feedback rows. `Flow.from_pending(flow_id)` always constructs a default `SQLiteFlowPersistence()`, so drift between save-time and resume-time breaks HITL resume with "No pending feedback found for flow_id" (AMP hit this).
2. `plan` (`@listen(ingest)` + `@human_feedback`) ‚Äî runs `PlannerCrew`, then pauses for human review. The `@human_feedback` decorator emits one of `approved | amend | rejected`.
3. `revise_plan` (`@listen("amend")`) ‚Äî folds the user amendments into the existing plan via the lighter `PlannerCrew().amend_crew()` (a single `amend_task` that edits only the work packages the feedback touches, instead of re-running the full `skeleton`‚Üí`expand`), then re-gates on another `@human_feedback`. **Recoverability invariant:** this method runs *during* `resume`, after CrewAI's `resume_async` has already cleared the pending-feedback row ‚Äî so it must NEVER raise. Any planner/`validate_plan` failure is caught and it falls back to the prior plan (annotated with an `open_question`) so the re-gate still fires and the job stays resumable. Raising here would leave no pending row and brick the job (`from_pending` would report "No pending feedback found"). See `_prior_plan_snapshot`.
4. `build` (`@listen("approved")`) ‚Äî chooses `build_dir` based on `plan.mode`. `patch_existing` writes into the resolved attached project root (`_resolve_patch_root`, supporting git clones and zip wrapper directories) and strips wrapper prefixes such as `inputs/<dir>/` from planned paths. `new_project` writes into `output/` and `git init`s it. It then loops over `plan.subtasks` running the Writer‚ÜíReviewer loop; `main.py::_build_subtask` returns a `ReviewResult`, and the build stops on the first unresolved non-passing subtask with `state.status = "failed"` and a `QAReport`.
5. `finalize` (`@listen(build)`) ‚Äî first runs the import-completeness pass and (for `new_project` RPA jobs) the packaging remediation pass: `runtime_qa.rpa_packaging_remediation_subtask(build_dir)` checks the Windows build kit (`.env.example`, `build.spec`, `build.ps1`/`build.bat`, `pyinstaller` dev dep) and, if anything is missing, synthesizes a stub `SubTask` that goes through the normal Writer‚ÜíReviewer loop before QA. Then runs deterministic integration QA (`ruff`, mypy symbol-drift checks, `.env.example` consistency checks, and pytest) over the workspace via `_run_final_qa`, which passes `require_installable=True` for `new_project` jobs ‚Äî `run_final_qa` provisions the generated project's own venv (`tools/project_env.ensure_project_env`, i.e. `uv sync` in the build dir) and fails QA with the sync output if the generated `pyproject.toml` can't install. Patch jobs run the full pytest suite whenever tests exist, and pytest still runs when lint/type gates fail so repair sees the real acceptance-test failures. If QA fails, it gives `WriterCrew.repair_crew()` one repair pass by default, reruns QA, then runs the domain architecture gate for new-project jobs via `run_full_architecture_gate(build_dir, plan)`. The gate is dispatched on `plan.domain` (a slug the planner emits, matching an activated architecture-defining skill); `runtime_qa._ARCHITECTURE_GATES` maps slugs to gate functions (currently only `"rpa"`). Projects with no domain ‚Äî or a domain without a registered gate ‚Äî finalize on deterministic QA alone. `finalize` captures a git diff for patch jobs, zips the current project even when QA failed, records history, and returns a completion payload with `qa_passed` as the safety signal. Failed QA payloads may still include `project_archive`, `zip_path`, `zip_url`, archive entries in `artifact_urls`, and `qa_report_markdown`. `on_rejected` and build-failure branches also call `history.record` so every terminated job is auditable.

State is a pydantic `CodebuilderState` (`schemas.py`) extending `FlowState`. `JobStatus` transitions: `pending ‚Üí planning ‚Üí awaiting_approval ‚Üí executing ‚Üí done|failed`.

**HITL wiring.** `feedback_provider.WebhookFeedbackProvider` POSTs to `$CODEBUILDER_APPROVAL_WEBHOOK` and raises `HumanFeedbackPending` to pause the flow. If the env var is unset it falls back to `ConsoleProvider` so the flow still works in a terminal. Resumes always go through `CodebuilderFlow.from_pending(job_id).resume(feedback)`. CrewAI's `resume_async` clears the pending-feedback row **before** running the routed `@listen` method and re-raises any non-`HumanFeedbackPending` exception ‚Äî so any work a resumed listener does (notably `revise_plan` re-running the planner) must be failure-safe or it bricks the job. A `@human_feedback` listener that itself produces output (like `revise_plan`) re-pauses and re-persists automatically when it requests the next round of feedback, so amend ‚Üí approve is a two-resume sequence: the first `resume("amend")` runs `revise_plan` and re-pauses; a second `resume("approve")` then routes to `build`.

### Crews (src/codebuilder/crews/)

Three `@CrewBase` classes, each with YAML-driven `agents.yaml` + `tasks.yaml` siblings:

- `PlannerCrew` ‚Äî one `planner` agent (Opus, `planning: true`). `.crew()` runs the first-plan path: a 2-task chain `skeleton_task` ‚Üí `expand_task` wired via CrewAI `Task(context=[‚Ä¶])`, whose result is the `Plan` from `expand_task`. `.amend_crew()` runs the revision path: a single `amend_task` (a plain method, NOT `@task`, so it's excluded from the auto-collected `self.tasks` used by `.crew()` ‚Äî same pattern as `WriterCrew.repair_crew`) that takes the prior plan + amendments and edits only the affected work packages. Both `expand_task` and `amend_task` carry the deterministic callable `guardrail=_require_nonempty_plan` (rejects empty/invalid plans so the crew retries in-loop ‚Äî addresses the empty-`subtasks` failure mode below; it also force-splits any work package over `MAX_FILES_PER_WORK_PACKAGE` by failing the guardrail with a "split this" message, so the planner re-bundles in-loop instead of letting `validate_plan` raise and crash `plan()`); `validate_plan` (`runtime_qa.py`) is still the authoritative post-kickoff check (1‚Äì24 work packages, each with 1‚Äì`MAX_FILES_PER_WORK_PACKAGE` (=8) non-empty `files`/`test_criteria`, no duplicate paths, no placeholder paths, no diagnostic-only RPA/code plans). The two structural caps `MAX_WORK_PACKAGES`/`MAX_FILES_PER_WORK_PACKAGE` live in `schemas.py` (single source of truth shared by the guardrail, `validate_plan`, and the planner prompt). Agent has only `FileReadTool` + `DirectoryReadTool` ‚Äî does not write files. Uses the `rpa` CrewAI skill. The `skeleton_task` prompt receives a `prior_history` markdown block assembled from the `project_history` SQLite table, and patch planning also receives the truncated preflight QA report when available. NOTE: a previous iteration routed `expand_task` to a Sonnet-backed `planner_lite` agent for cost/latency; it was reverted after Sonnet's `expand_task` returned an empty `Plan.subtasks` list in production ‚Äî hence the guardrail. If reintroducing Sonnet on the planner side, keep the guardrail and fall back to the Opus agent on failure.
- `WriterCrew(workspace_dir)` ‚Äî one `writer` agent. `.crew()` runs `write_task` for planned files; `.repair_crew()` runs `repair_task` after failed final QA and can patch all files implicated by the QA report in one `CodeBundleArtifact`. Scoped to `workspace_dir` via the three `Workspace*Tool`s. Deterministic review forbids placeholder/TODO/`...` stubs.
- `ReviewerCrew(workspace_dir)` ‚Äî `.crew()` runs per-file review fallback, `.qa_crew()` provides LLM QA diagnostics, and `.architecture_gate_crew()` is the domain architecture acceptance gate for new projects (driven by `plan.domain`). Reviewer/QA agents use both the `rpa` and `code-review-gate` skills. The deterministic final-QA path runs `ruff`, mypy symbol-drift checks, `.env.example` consistency checks, and pytest before any LLM diagnostics.

The Writer‚ÜîReviewer loop in `main.py::_build_subtask` retries up to `CODEBUILDER_MAX_SUBTASK_RETRIES + 1` times per subtask (default 4 total attempts), feeding deterministic issues plus reviewer issues/suggestions back into the writer's inputs. `_build_subtask` returns the final `ReviewResult`; callers stop later subtasks after an unresolved non-passing subtask, then let `finalize()` run final QA, zip the current workspace, and report the failed subtask in `qa_report_markdown`. Final QA repair is controlled separately by `CODEBUILDER_MAX_FINAL_QA_REPAIRS` (default `1`). Skipped lint/tests are failures. New-project jobs whose plan declares a `domain` registered in `_ARCHITECTURE_GATES` must also pass that gate before status becomes `done`.

### Schemas (src/codebuilder/schemas.py)

All crew outputs go through pydantic models: `Plan`, `SubTask`, `CodeBundleArtifact`, `CodeArtifact`, `ReviewResult`, `QAReport`, `ProjectArchiveRef`, `Attachment`. When adding fields, the YAML task description's `Rules:` section and `expected_output:` typically need matching updates, since the agent produces JSON against this schema.

`CodebuilderState` also carries `project_name` (user-supplied, for display and `new_project` keying), `project_key` (derived stable identifier used for the `project_history` table), and optional `preflight_qa_report` (deterministic patch QA captured before planning). `Plan.project_name` stays as the planner's agent-produced display name ‚Äî distinct from the state field.

### Per-project history (src/codebuilder/history.py)

Standalone SQLite table `project_history` at `$CODEBUILDER_HISTORY_DB` (defaults to `./data/codebuilder_history.db`), independent of CrewAI's flow persistence (only the HITL pending-feedback rows that CrewAI's auto-fallback writes live there). One row per terminated job keyed by `(project_key, job_id)`. Public API:

- `canonicalize_git_url(url)` ‚Äî reduces git URLs to `host/org/repo` (handles HTTPS/SSH/scp-style/casing/`.git` suffix).
- `project_key_from(state)` ‚Äî derives the stable key: canonical git URL from the first `Attachment` with `kind == "git"`, else slug of `project_name`, else `""` (main.py falls back to `flow_id`).
- `record(state)` ‚Äî upserts a row on `finalize` / `on_rejected` / build-failure. Serialises `Plan`, `QAReport`, artifact `file_path`s, aggregated reviewer issues, and git diff (for patch jobs).
- `summarize_for_planner(project_key, limit=3)` ‚Äî compact markdown of the last N runs (date, mode, status, files touched, top reviewer issues, QA notes) fed into `{prior_history}` in the planner task YAML. Returns `""` when there's no history.

History is observability, not correctness ‚Äî all call sites wrap `history.record(...)` in `try/except` so a DB failure never breaks the flow.

### Tools (src/codebuilder/tools/)

- `workspace_tool.py` ‚Äî `WorkspaceRead/Write/List` all share `resolve_within()` which enforces that agent-supplied relative paths cannot escape the job workspace. All file I/O from agents goes through these; never give agents raw `FileReadTool` pointed at a real filesystem path.
- `lint_runner_tool.py` ‚Äî runs `ruff check` / `pytest -q` inside the workspace (120s / 300s timeouts) **with the workspace project's own interpreter**: both tools call `ensure_project_env()` then execute `<project_python> -m ruff|pytest`, falling back to the orchestrator's interpreter when the module isn't installed in the project venv. This is what lets a generated src-layout package import in its own tests ‚Äî running pytest from codebuilder's venv (where the generated package and its deps are never installed) was the root cause of every skill-compliant RPA project dying at pytest collection. Explicitly-targeted non-Python files (`build.spec`, `*.ps1`, `*.md`, ...) return `"PASS"` without invoking ruff ‚Äî explicit-file ruff lints anything passed (PyInstaller spec files hit F821 on injected globals), whereas directory scans already skip non-`.py`. Returns `"PASS"` on clean exit, otherwise the captured stdout+stderr.
- `project_env.py` ‚Äî `ensure_project_env(workspace_dir)` runs `uv sync` in the build dir (no-op without a `pyproject.toml`, when `uv` is missing, or when `CODEBUILDER_PROVISION_PROJECT_ENV=0`); returns `""` on success or the sync error text. A sha256-of-pyproject marker in `.venv/` skips re-syncs until the pyproject changes. `project_python(workspace_dir)` returns the project venv's interpreter when it exists, else `sys.executable`. `VIRTUAL_ENV` is stripped from the subprocess env so uv resolves the *project's* `requires-python` (it will download e.g. CPython 3.13 even though codebuilder runs on 3.11).
- `git_tool.py` ‚Äî thin `gitpython` wrappers: `clone`, `init_and_commit`, `diff`. Used for patch-mode diff extraction and new-project baselines. Both commit/diff paths write `_HARNESS_EXCLUDES` (`.venv/`, `uv.lock`, caches) to `.git/info/exclude` before `git add -A`, so QA provisioning artifacts never enter baselines or patch diffs.
- `attachment_tool.py::materialize()` ‚Äî decodes the `Attachment` list (git URLs cloned, zips extracted, PDFs saved + text-extracted via `pypdf`, images saved raw) into `workspace/inputs/`. Patch jobs then resolve the real project root with `_resolve_patch_root`: git attachments usually land at `inputs/repo`, while zip attachments may have an extra wrapper directory that must be stripped from planned paths.

### Skills

Methodology and checklist guidance lives in CrewAI skills under `src/codebuilder/skills/`: `rpa` carries the canonical RPA onboarding guide (Portuguese, full process/architecture/SOLID/TDD/folder-layout standard) and `code-review-gate` carries the domain-agnostic acceptance criteria for generated projects. Each crew file pre-discovers the skills dir into a `_SKILLS` dict at module load and passes specific entries inline (`skills=[_SKILLS["rpa"]]`), the same way tools are passed. New domain skills (e.g. `python-package`, `flask-api`) drop into `skills/` and get attached by name. There is no separate skill loader and no `knowledge/` directory ‚Äî `Agent.skills` accepts pre-loaded `Skill` objects directly and `Agent.set_skills` activates them with proper event emission. Skill slugs are also the keys used by the architecture-gate registry (`_ARCHITECTURE_GATES` in `runtime_qa.py`) when the planner emits a matching `plan.domain`.

## Conventions specific to this repo

- Workspace root is `$CODEBUILDER_WORKSPACE_ROOT` (defaults to `./workspaces`). Each job lives in `<root>/<flow_id>/` with `inputs/` and `output/` subdirs. Never let tools touch anything outside this root.
- Cross-run context comes from the `project_history` SQLite table. Do not mutate `CREWAI_STORAGE_DIR` at runtime (HITL resume depends on it being stable).
- The stable project identifier is `state.project_key`, not `state.project_name`. Always derive via `history.project_key_from(state)` ‚Äî never reconstruct ad-hoc from the name (it loses the git-URL canonicalisation that makes two users hitting the same repo share context).
- Kickoff payloads MUST use `session_id` (not `id`) for the caller's session identifier. See the `ingest` note for why ‚Äî overriding `state.id`/`flow_id` strands AMP OTel traces. The HITL feedback webhook and the progress webhook both emit a `session_id` field; the frontend correlates incoming events on `session_id` first, with `flow_id`/`job_id` as fallbacks.
- Default models and planning settings live in each crew's `agents.yaml`. Edit YAML for agent model changes. The only remaining model env override is `CODEBUILDER_GUARDRAIL_LLM` for the `@human_feedback` classifier.
- Planner YAML sets `planning: true` on `planner` (used by the planning tasks). The writer runs without `planning`/`reasoning` ‚Äî it follows an explicit, fully-specified subtask and doesn't need a reasoning pre-pass.
- When adding a task that returns structured data, set `output_pydantic=<Schema>` and, where correctness matters, a `guardrail=...` on the `Task`. Prefer a deterministic callable guardrail (no extra LLM call/latency) over a string one ‚Äî see `_require_nonempty_plan` on `expand_task`/`amend_task` in `planner_crew.py`. Callable guardrails must take exactly one positional arg and return `(bool, Any)`; returning a non-`str`/non-`TaskOutput` value on success leaves the `TaskOutput.pydantic` untouched.
- `_planner_inputs()` in `main.py` is the single source of truth for the dict passed to *both* `PlannerCrew().crew()` (first plan) and `.amend_crew()` (revisions). Add new template variables here **and** in `crews/planner_crew/config/tasks.yaml` together ‚Äî the YAML uses `{var_name}` interpolation and will raise on missing keys (a task may reference a subset of the keys; extra keys are ignored).
- `preflight_qa_report` is planner context only. Keep `_run_patch_preflight_if_available`, `_planner_inputs()`, and `crews/planner_crew/config/tasks.yaml` aligned so patch plans can use concrete failures without exposing preflight QA as a runnable deliverable.
- Kickoff payloads accept an optional top-level `project_name` (required for useful `project_history` keying in `new_project` mode; `patch_existing` jobs derive the key from the git attachment URL instead).
- QA runs in the generated project's environment, not codebuilder's. Any QA-adjacent change must keep the `ensure_project_env` ‚Üí `project_python` path intact; never run ruff/pytest against a generated project with bare `sys.executable` (the generated src-layout package is not importable there). `CODEBUILDER_PROVISION_PROJECT_ENV=0` disables provisioning (tools then fall back to the orchestrator interpreter ‚Äî useful for unit tests, wrong for real jobs).
- Archive fields are not success-only. Failed QA completion payloads can include `project_archive`, `zip_path`, `zip_url`, and project-archive entries in `artifact_urls`; callers must use `qa_passed` / `qa_report.passed` to decide whether the archive is verified.
- Plan validation rejects placeholder file names and diagnostic-only RPA/code plans. Do not reintroduce work packages whose output is "inspect later", markdown-only diagnostics, or files named with `TBD`, `PLACEHOLDER`, or `*_TO_BE_DETERMINED_*`.
- Planner question hygiene: `skeleton_task` is prompted to prefer recorded `assumptions` over `open_questions` (max 3, only for genuinely blocking decisions, empty list otherwise). Keep that rule when editing the planner YAML ‚Äî first-turn question spam was an explicit user complaint.
